## Findings (template review)

### 🔴 Finding 1 — Deploy-time blocker (most likely root cause)
Adding `!ImportValue ${StackName}-permission-invalidation-queue-url` to `Globals` makes the **entire DynamicTables stack fail to deploy** unless the Access stack is already deployed and exporting that value in the same env. Access/ is new/untracked → not deployed to dev → CFN error `No export named ...-permission-invalidation-queue-url found`, rollback. Net: new `updateTable.ts` never lands → PATCH hits old code → consistent with "no logs / nothing new." Same blocker hits Folders + Assistant. No `DependsOn` enforces Access→consumers ordering (independent module stacks).

### 🟠 Finding 2 — Runtime IAM gap (latent; bites non-superusers post-deploy)
No DT role grants `dynamodb:GetItem/Query` on `AccessGrant`/`TeamUser` or `sqs:SendMessage` on the invalidation queue. Non-bypass roles → AccessDenied in PermissionResolver + silent invalidation failure. Superuser token masks it.

### 🟡 Finding 3 — Blast radius
Import in `Globals` → all ~40 functions (incl. pure reads) hard-depend on the export. One missing export bricks the module.

## Decisions Made
- Root cause hypothesis prioritized: **deploy-time import blocker (Finding 1)**, pending confirmation via `aws cloudformation list-exports`.
