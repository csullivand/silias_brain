# Session: 2026-07-14 — CASL IAM grants for Folders & DynamicTables (admin delete fix)

## Context
- **Branch:** develop
- **Working directory:** /Users/sulli/Projects/silia/06-11-25/silia/
- **Ticket/Issue:** Prod error `createFolder AccessDeniedException` + requirement "Debo poder eliminar el último folder, table o agent de una cuenta"
- **Goal:** Let non-superuser admins create/list/delete folders and tables (and rows/columns/connections/etc.). Root cause was an IAM gap, not a "last item" rule.

## Root Cause (the important finding)
Reported symptom: *"superuser can but admin doesn't"* and *"can't delete the last folder/table/agent"*.

Real cause: the CASL permission check does a **`dynamodb:Scan` on the `${StackName}-role` table**, but the Lambda exec roles were **not granted access to the CASL permission tables** (`-role`, `-permission`, `-resource`).

Call chain:
- Handler calls `assertPermission(event, '<perm>')` (`shared/middleware/requirePermission.ts`)
- -> `defineAbilitiesForRole(role)` (`shared/config/permissions.ts:81`)
  - SUPERADMIN/SUPERUSER short-circuit to `can('manage','all')` at line 84 -> **no DB read** -> works
  - Any other role -> `resolveRoleId(role)` (line 99) -> `new RoleModel({}).findAll()`
    -> `super.scan({ TableName: process.env.ROLE_TABLE || \`${STACK_NAME}-role\` })` (`Roles/domain/models/Role.model.ts:196`) -> **AccessDenied**

That asymmetry (superuser bypasses DB, admin hits the Scan) is exactly the reported behavior.

There is **NO code anywhere** (backend or frontend) that forbids deleting the last folder/table/agent. Frontend `useHomeObjects.ts:176` just shows the empty state after the last delete. So the requirement was really "unblock admins", satisfied by fixing IAM.

## CASL access patterns (what the exec roles need to read)
| Table | DynamoDB op | Source |
|-------|-------------|--------|
| `${StackName}-role` | `Scan` | `RoleModel.findAll` via `resolveRoleId` |
| `${StackName}-permission` (+ `/index/*`, GSI `roleId-resourceId-index`) | `Query` | `PermissionModel.findAllByRoleId` |
| `${StackName}-resource` | `GetItem` | `ResourceModel.findByIds` -> `findById` |

(`ResourceModel.findAll` Scan exists but is cold-start-only validation, non-blocking, so `resource` needs only `GetItem`.)

## Reference pattern followed
Checked all templates granting these tables: `Roles/` (owner), and **`Assistant/` + `Access/` already grant them**. Copied the canonical least-privilege 3-statement block from `Assistant/infrastructure/aws.template.yml:1780`:
```yaml
# CASL permission resolution (shared/config/permissions.ts) — read-only
- Effect: Allow
  Action: [dynamodb:Scan]
  Resource: !Sub "arn:aws:dynamodb:${AWS::Region}:${AWS::AccountId}:table/${StackName}-role"
- Effect: Allow
  Action: [dynamodb:Query]
  Resource:
    - !Sub "arn:aws:dynamodb:${AWS::Region}:${AWS::AccountId}:table/${StackName}-permission"
    - !Sub "arn:aws:dynamodb:${AWS::Region}:${AWS::AccountId}:table/${StackName}-permission/index/*"
- Effect: Allow
  Action: [dynamodb:GetItem]
  Resource: !Sub "arn:aws:dynamodb:${AWS::Region}:${AWS::AccountId}:table/${StackName}-resource"
```
No `kms:` grant needed — CASL tables use the AWS-owned key (no `SSESpecification`/CMK in `Roles/infrastructure/aws.template.yml`).

## Auth mechanism per object (why agents were fine)
- **Folder** delete/create/list -> `assertPermission(...)` -> CASL Scan -> was broken -> fixed.
- **Table** (+rows/columns/docs/connections) -> `requireTableAccess` -> `assertPermission` -> CASL Scan -> was broken -> fixed.
- **Agent/chatbot** delete -> `assertRole(event, ELEVATED_ROLES)` (JWT claims only, no DB) -> never affected.
