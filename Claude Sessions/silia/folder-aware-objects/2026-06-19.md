## Architecture Decision
- **Chose Option 3**: `folderId` on each model + denormalized `itemCounts` counter on FolderModel
- Rejected junction table (Option 2) — adds dual-write complexity without proportional benefit
- 3 parallel DynamoDB queries is fast enough (<15ms with Promise.all)
- Counters are atomic (DynamoDB `if_not_exists + ADD`), fire-and-forget with try/catch

## Key Design Details

### folderId field
- `parentFolderId` on Folders uses `'ROOT'` sentinel (GSI key, can't be null)
- `folderId` on Chatbots/Tables uses `null` for root (not a GSI key, null is fine)
- Conditional save: `...(this.folderId !== undefined ? { folderId: this.folderId } : {})`

### itemCounts
- `{ folders: number, agents: number, tables: number }` on FolderModel
- Atomic increment/decrement via `FolderModel.incrementItemCount(folderId, field, delta)`
- Used by listFolders to skip unnecessary queries (if agents=0, don't query chatbots)

### Unified listing
- `GET /folders?parentFolderId=ROOT` → root folders + root agents + root tables
- `GET /folders?parentFolderId=<uuid>` → subfolders + agents + tables in that folder
- `GET /folders` (no param) → all folders in account (original behavior)
- Each item has `type: 'folder' | 'agent' | 'table'` discriminator
- Sorted by createdAt descending

- Agent create (`Post/index.ts`) → increment agents

- Agent PUT folderId change (`Put/index.ts`) → decrement old, increment new
- Table create (`createTable.ts`) → increment tables

- Folder create (`createFolder.ts`) → increment parent folders
- Folder delete (`deleteFolder.ts`) → decrement parent folders

### Step 7: Migration script

- Idempotent, skips folders with correct counts

- **File:** `Folders/application/Folders/get/listFolders.ts`
- Extended to return folders + agents + tables when `parentFolderId` is specified
- Uses folder's `itemCounts` to skip queries when count is 0
- For ROOT level, always runs all 3 queries (no parent folder to check counts)
