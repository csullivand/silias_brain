# 2026-07-14

## Main work: CASL IAM grants for Folders & DynamicTables
Fixed the root cause behind "admin can't delete the last folder/table/agent" and the prod `createFolder AccessDeniedException`. It was never a "last item" rule — non-superuser admins were blocked because the Lambda exec roles lacked read access to the CASL permission tables (`-role`/`-permission`/`-resource`) that `assertPermission` scans. Superusers short-circuit to `manage:all` and skip the DB, hence "superuser can but admin doesn't".

Granted the canonical least-privilege CASL block (Scan role / Query permission+index / GetItem resource; no KMS) — copied from how `Assistant/` and `Access/` already do it — to:
- Folders: FolderReadRole, FolderWriteRole
- DynamicTables: 10 CASL roles (table/column/row read+write, document, agent-connection, filterbar-config, assistant-suggestions)

Infra-only, cfn-lint clean, not yet committed.

## Decisions
- Fix at IAM layer (not code), matching existing modules.
- Widened beyond delete to the whole CASL surface so admins aren't half-broken.

## 🐛 BUG POST-DEPLOY + FIX (2026-07-15)

**Síntoma:** los contadores de `GET /teams` (folderCount/agentCount) daban **siempre 0** en dev, aunque hubiera datos.

**Causa raíz (de los CloudWatch logs):**
```
AccessDeniedException: role/dev-app-silia-com-teams-get-all-role is not authorized
to perform: dynamodb:Query on table/dev-app-silia-com-AccessGrant/index/subjectId-objectType-index
  at ... findBySubjectId
```
`countTeamResources` → `AccessGrantModel.findBySubjectId` → **AccessDenied** → el try/catch fail-open lo tragaba → devolvía {folderCount:0, agentCount:0}.

**El error del PR de counters (SL-1281):** el IAM de AccessGrant se agregó al **rol equivocado**. La función que sirve `GET /teams` es `GetAllTeamsFunction` (Handler getAllTeamsMin.handler, FunctionName `-teams-get-all`) → usa **`TeamsGetAllRole`** (RoleName teams-get-all-role). Pero el IAM se le puso a `TeamsGetByAccountRole` (otra función). `TeamsGetAllRole` tenía Team/TeamUser/Account/User pero NO AccessGrant.

**Fix:** agregado `dynamodb:Query` sobre `-AccessGrant` + `/index/*` a **`TeamsGetAllRole`**.
- Rama: `feat/SL-1281-embed-teams-members`, commit `1177e8406`, pusheado.
- ⚠️ Es INFRA → solo aplica al **desplegar** el módulo Teams. Hasta entonces sigue en 0.

**`/management` (contadores de usuarios):** OK — `UserListOrgUsersRole` (rol real de listOrgUsers) SÍ tiene AccessGrant/Team/Folders/Chatbot. No afectado.

**Lección:** al agregar IAM cross-módulo, verificar el ROL REAL de la función (por RoleName / FunctionName), no asumir por el nombre del handler. Este mismo error casi lo repito con el embed de `members` (que lee TeamUser+User) — pero `TeamsGetAllRole` ya tenía esos, así que ese feature sí funciona.

**Pendiente menor:** quedó AccessGrant sin uso en `TeamsGetByAccountRole` (el que se puso mal); inofensivo, limpiar opcional.

**Semántica recordatorio:** folderCount/agentCount de un team = filas de AccessGrant con subjectId=teamId (grants directos). Los teams NO son dueños de folders; se cuentan los grants. Sin grants → 0 (legítimo). Los grants se crean con POST /access/grants (Feature 5).
