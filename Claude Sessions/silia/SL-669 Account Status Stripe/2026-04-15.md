# SL-669 Account Status Stripe Pause/Resume ‚Äî 2026-04-15

## Objective
When a Super Admin deactivates an account via `PATCH /{id}/status`, Stripe subscriptions for all chatbots on that account should be paused (no invoices generated). When reactivated, subscriptions should resume.

## Problem
`AccountStatusService.changeStatus()` deactivated chatbots (`status=0`) but never touched Stripe. Subscriptions kept running, invoices kept generating, clients kept getting charged for inactive accounts.

---

## Analysis & Decisions

### What already existed (no changes needed)
- `PATCH /{id}/status` endpoint ‚Äî superuser-only, with `requireSuperuserMiddleware`
- `AccountStatusService.changeStatus()` ‚Äî handles status transitions, chatbot deactivation/activation, audit logging via `AccountStatusAuditModel`, email notification via `accountStatusChange` template
- `AccountStatusAudit` DynamoDB table ‚Äî logs `accountId`, `performedByUserId`, `previousStatus`, `newStatus`, `timestamp`
- Frontend: status dropdown + confirmation modal in `AccountManagement.tsx` (lines 2147-2174)
- `AccountStatusUpdate` Lambda already has `STACK_NAME` env var ‚Üí can access Stripe secret via SSM

### Branch strategy
- Created `feat/SL-669-account-status-change` from `develop`
- Merged `feat/SL-651-activar-desactivar-agente` into it ‚Äî this branch had `pauseSubscription`/`resumeSubscription` already implemented in `BillingService` and `StripeGateway` (using Stripe's `pause_collection: { behavior: 'void' }`)

### Cancel vs Pause decision
- **Chose pause** ‚Äî subscription stays alive in Stripe, no invoices generated (voided), easy resume on reactivation
- Cancel would require creating new subscriptions on reactivation ‚Äî too disruptive
- `cancelSubscription` reserved for permanent removal (e.g. deleting a chatbot)

- `chatbot-status-pause-resume-plan.md` ‚Äî individual chatbot toggle (SL-651, implemented on its branch)
- `billing-changes-plan.md` ‚Äî account billing features (already done)
- `subscription-schedules-plan.md` ‚Äî future `effectiveFrom` handling (already done)
- None existed for account-level status ‚Üí Stripe. Created `account-status-manual-change-plan.md`

## Dunning Process Planning

### Ticket
BILLING: Proceso de dunning automatico ‚Äî automated 7-day retry process when payment fails, escalating emails, account suspension on day 7.

### Key Decision: Use Stripe Smart Retries
Instead of building a custom cron + DynamoDB dunning state table, we leverage Stripe's built-in dunning:
- Stripe handles retry timing (ML-optimized Smart Retries)
- Stripe tracks attempt_count on each invoice
- Stripe transitions subscriptions: active ‚Üí past_due ‚Üí unpaid
- We react to webhooks and handle notifications + suspension

### What Already Existed
- Daily cron deactivateOverdueChats (2AM UTC) ‚Äî only fires at >7 days, no retries
- handlePaymentFailure webhook ‚Äî sends single paymentFailed email, no attempt tracking
- getWarningMessage() with day 3/5/7 tiers ‚Äî dead code (only called after day 7)
- DunningAlertEmail.tsx and PaymentAttemptFailedEmail.tsx ‚Äî designed but not wired
- retryPendingPayments() ‚Äî only fires when new card added, not on schedule
- David Luna (davidluna@creai.mx) created the original Billing module and cron

Saved at zzzDocs/dunning-stripe-smart-retries-plan.md

### Dunning Flow
- Day 0: Payment fails ‚Üí Stripe fires invoice.payment_failed (attempt 1) ‚Üí we send gentle email
- Days 1-3: Stripe retries ‚Üí each failure fires invoice.payment_failed ‚Üí escalating emails
- Days 4-5: More retries ‚Üí stronger warnings
- Day 6: Pre-suspension alert to customer + internal alert to commercial team
- Day 7: Final failure ‚Üí Stripe marks subscription unpaid ‚Üí customer.subscription.updated fires ‚Üí we call AccountStatusService.changeStatus('inactive') (SL-669) ‚Üí all chatbots paused, all Stripe subs paused, audit log, notifications
- Recovery: Customer adds card / retries ‚Üí invoice.paid ‚Üí activateAssistant reactivates chatbot
