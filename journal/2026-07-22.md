# 2026-07-22 — Feature 0 CASL: unblocking permission-gated endpoints

Big debugging day on the Feature 0 (SL-1282) permissions rollout. Full detail: [[Claude Sessions/silia/feature-0-casl-iam-grants/2026-07-22]].

## Main changes
- **CASL IAM grants** applied to all 12 dedicated-role modules (~91 roles) — read on `-role`/`-permission`/`-resource` + `/index/*`. Committed on `feat/SL-1282-permissions`.
- **Seeds run:** STAGING (Group A+B) and DEV (Group B). DEV now role=7, resource=190, permission=572.
- **Workflows/Integrations CORS fix:** `AddDefaultAuthorizerToCorsPreflight: false` (OPTIONS preflight was 401, masqueraded as CORS).
- **webpack.config.js:** prod `devtool` → `false` (source maps blew Lambda past 250MB unzipped).
- **Logger:** `shared/middleware/requirePermission.ts` migrated console.* → Pino `securityLog`.
- **getWebChatConfig.ts:** removed a wrongly-added `assertPermission` guard — `GET /chatbot/{id}` is a PUBLIC endpoint (no authorizer).
- **SSO:** local `~/.aws/config` role → `silia-platform-sandbox`.

## Decisions
- User was right to reject 'stale deploy' — CI (`deploy-service.yml`) already `nx reset`s + clean-builds, so deployed code IS current. The 403s were real bugs: missing authorizer on a public route + a guard that needs a role that never arrives.
- Public config endpoints must NOT carry `assertPermission` (no authorizer → no role → denies everyone).
