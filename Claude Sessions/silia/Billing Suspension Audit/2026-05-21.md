### Step 5: Identified critical bug — blind reactivation
- **What:** `updateChatbotsStatus(accountId, 1)` in `AccountStatusService.ts:70-91` reactivates ALL chatbots, including ones the user had manually disabled before suspension
- **Root cause:** No distinction between 'suspended by dunning' vs 'disabled by user choice'
- **Solution designed:** Use existing `suspensionReason` field on Chatbot model as the filter

### Step 6: Designed suspensionReason-based fix
- **Approach:**
  - Add optional `reason?: 'dunning' | 'manual'` to `ChangeStatusInput` interface
  - Pass it through to `updateChatbotsStatus()`
  - **Deactivating (status=0):** Only touch chatbots with status=1, set `suspensionReason = reason`
  - **Reactivating with reason (status=1):** Only restore chatbots where `suspensionReason === reason`
  - **Reactivating without reason (admin manual):** Restore all (current behavior preserved)
- **Callers analysis:**
  - `Accounts/.../updateAccountStatus.ts:84,121` — admin manual, no reason → touches all (correct)
  - `BillingService.ts:185` — safety-net cron → should pass `reason: 'dunning'`
  - `StripeWebhookHandler.ts:897` — webhook dunning → should pass `reason: 'dunning'`
  - `StripeWebhookHandler.ts:553` — reactivation by payment → should pass `reason: 'dunning'`
- **Status:** Design approved by user, pending implementation
