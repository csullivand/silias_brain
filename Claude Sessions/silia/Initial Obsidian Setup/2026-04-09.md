## Checkpoint ŌĆö 2026-04-13: Conversation Close Research

### Task
User asked to expose an endpoint for closing conversations from n8n.

### Research: How conversations are closed in the codebase

#### Model (`Messages/domain/models/ChatConversation.model.ts`)
- `status`: `'active' | 'closed_by_inactivity' | 'closed_by_agent' | 'closed_by_escalation_resolution'`
- `conversationState`: `'open' | 'require_takecontrol' | 'in_progress' | 'resolved'`
- `closedReason`: string
- `closedAt`: number (timestamp)
- Two separate concepts: `status` = lifecycle, `conversationState` = workflow state

#### 3 existing close paths
1. **`shared/services/ConversationCoordinator.class.ts:218`** ŌĆö `closed_by_inactivity` on inactivity threshold exceeded
2. **`Assistant/domain/services/EscalationRecordService.ts:632`** ŌĆö `closed_by_escalation_resolution` when agent resolves escalation
3. **`Messages/application/orphanConversationCleanup/index.ts:56`** ŌĆö `closed_by_inactivity` scheduled cleanup

#### Key finding: `closed_by_agent` is never used in backend
- Exists in type definition and in `ChatConversationMonthlyCount` billing tracking
- But no backend code ever sets it

#### Existing endpoints
- `PUT /conversation/{id}` (`putConversation.ts`, template line 229) ŌĆö generic update, no close logic
- `PUT /conversations/{id}/resolve` ŌĆö frontend uses for `conversationState: 'resolved'` (different from closing)
- Frontend `conversationStateService.ts` calls: claim, release, resolve, reassign, requestTakeControl

#### Close logic that a proper endpoint needs
1. Set `status: 'closed_by_agent'`, `closedAt`, `closedReason`
2. Update monthly billing count (`ChatConversationMonthlyCount.incrementOnClose`)
3. Trigger summary generation (`ConversationMemoryService.scheduleSummaryGeneration`)
4. Publish `conversation.updated` WebSocket event

### Decision pending
Waiting for user to confirm: build dedicated `PUT /conversation/{id}/close` or reuse existing putConversation.

## Checkpoint ŌĆö 2026-04-13: Deep Investigation & Architecture Decision

### Investigation: What's reusable for close endpoint

#### Reusable methods (all standalone callable)
- `ChatConversation.model.ts:342` ŌĆö `conversation.update(Partial<ChatConversationI>)` ŌĆö DynamoDB write
- `ChatConversationMonthlyCount.model.ts:165` ŌĆö `incrementOnClose(chatbotId, cycleStart, reason)` ŌĆö billing update
- `ConversationMemoryService.class.ts:88` ŌĆö `scheduleSummaryGeneration(conversationId)` ŌĆö fire-and-forget
- `shared/websocket/WebSocketEventService` ŌĆö `publish(createWebSocketEvent(...))` ŌĆö same pattern as putConversation.ts:98-122

#### Key finding: `updateMonthlyCountOnClose` in ConversationCoordinator is PRIVATE
- Can't call it from outside
- But `ChatConversationMonthlyCount.incrementOnClose()` is PUBLIC and standalone ŌĆö call that directly

#### Existing close pattern (ConversationCoordinator:217-224)
```typescript
await conversation.update({ status: 'closed_by_inactivity', closedAt: now, closedReason: '...' });
await this.updateMonthlyCountOnClose(chatbotId, conversation.billingCycleStart, 'closed_by_inactivity');
ConversationMemoryService.getInstance().scheduleSummaryGeneration(conversation.id);
```

### Architecture decision: Messages/ module
- **Why Messages/:** ChatConversation model lives there, putConversation already there, orphanCleanup already there
- **Not Workflows/:** n8n is the caller, not the owner. Frontend might also use this endpoint later.
- **Not Assistant/:** Assistant handles orchestration, not data lifecycle

### Plan
1. Create `Messages/application/closeConversation/closeConversation.ts` (~80 lines)
2. Add Lambda + `PUT /conversation/{id}/close` to `Messages/infrastructure/aws.template.yml`
3. Handler calls 5 existing methods ŌĆö no new business logic
4. Env vars: `SHARED_WEBSOCKET_ENDPOINT` (already in Globals)
