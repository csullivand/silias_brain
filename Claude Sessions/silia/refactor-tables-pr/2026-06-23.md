## Issues Found & Fixed This Session

### 1. encoder.json + vocab.bpe Missing (RUNTIME CRASH)
- **Symptom:** 502 Bad Gateway on dev deployed + local. Lambda crashes at init: `ENOENT: no such file or directory, open '/var/task/encoder.json'`
- **Root cause:** DynamicTables and Folders handlers now import `ChatbotModel` (for ownership validation in requireTableAccess and unified listing in listFolders). ChatbotModel pulls in the OpenAI tokenizer which requires `encoder.json` and `vocab.bpe` at runtime.
- **Why it wasn't needed before:** Old code used `requireChatbotAccess` (lightweight, no ChatbotModel import). The refactor introduced cross-module ChatbotModel imports for the first time.
- **How webpack handles it:** `webpack.config.js` has `findModelFiles()` that searches each module's `application/` directory for `encoder.json`, `vocab.bpe`, `.dat`, `.zip` and copies them via CopyWebpackPlugin. But only searches WITHIN the module — not across modules.
- **Fix:** Copied `encoder.json` and `vocab.bpe` from existing modules (Billing/Assistant) into `DynamicTables/application/` and `Folders/application/`. Same pattern every other module follows.
- **All modules verified:** Every module that imports ChatbotModel has both files.

### 2. assertPermission Blocking Locally

## Key Learning: Cross-Module Imports and Webpack

When a Lambda handler imports a model from another module (e.g., DynamicTables importing ChatbotModel from Assistant), the webpack bundle includes the FULL dependency tree of that model. If any dependency reads files from disk at startup (like the OpenAI tokenizer), those files must be present in the source module's `application/` directory for webpack's CopyWebpackPlugin to include them.

**Pattern:** If you add a new cross-module import to a handler, check if the imported module's dependency chain needs `encoder.json`, `vocab.bpe`, or other data files. Copy them to your module's `application/` directory.
