## Topic 1: Multi-select Filter Fix (DynamicTables)

### Problem
Multi-select column filters don't match rows. Filter sends option `value` ('opcion_1') but DynamoDB stores option `label` ('Opcion 1').

### Root Cause
- `ColumnOptionsModal.tsx` generates values via `generateOptionValue(label)` → lowercase + underscores
- `MultiSelectCell` saves labels to DynamoDB (correct)
- `FiltersModal.tsx` Select sends `o.value` (wrong) instead of `o.label`
- `RowDetailModal.tsx` Select also sends `o.value` when editing

### Fix Applied
- `FiltersModal.tsx:471` — changed `value: o.value` → `value: o.label` for multi-select filter Select
- `RowDetailModal.tsx:289` — same fix for detail modal multi-select editor

### Status
Changes applied but FiltersModal was significantly refactored (AdvancedFilters component). Need to verify fix landed in correct location in new code.

## Topic 2: Chatbot Deactivation Investigation (Prod)

### Problem
Multiple chatbots across different accounts getting deactivated unexpectedly in production.

### Investigation
1. `deactivateOverdueChats` cron — **broken** (`HandlerNotFound: deactivateOverdueChatsMin.handler`), not the cause
2. `AccountStatusAudit` table — **empty**, ruling out `AccountStatusService.changeStatus`
3. Checked PUT /chatbot Lambda logs — config saves with no `status` field
4. Checked `updateAt` timestamps — deactivation happened weeks before recent PUT calls
5. One chatbot's `updateAt` (1780548655 → June 4 04:50 UTC) matches cron schedule exactly, but cron was already broken by then

### Key Findings
- `ChatbotModel` constructor defaults `status` to 0 when undefined (line 517)
- `Object.assign` with `status: undefined` would overwrite to undefined (but Zod strips missing fields)
- No `suspensionReason` on affected chatbots → rules out dunning
- Could not determine root cause — logs don't go back far enough (May 27)

### Recommendation
Add audit logging to the PUT /chatbot endpoint when status changes, so future deactivations are traceable.

## Topic 3: Folders CRUD — ROOT Fix

### Problem
`ValidationException: Type mismatch for Index Key parentFolderId Expected: S Actual: NULL` when creating root-level folders. DynamoDB GSI keys cannot be null.

### Fix Applied
- Replaced `null` with `'ROOT'` sentinel in 4 locations:
  - `createFolder.ts:63`
  - `FolderService.ts:76,85` (moveFolder)
  - `Folder.model.ts:78` (constructor)
- `findRootFolders` checks both `'ROOT'` and `null` for backward compat
- Fixed `deleteFolderTree` move-to-parent: added `isRoot` check instead of truthiness check on parentFolderId
- Added structured logging with Pino (`getLambdaLogger` / `createLogger`) to all handlers and FolderService
- Removed raw `JSON.stringify(event)` logs (JWT exposure risk flagged in PR review)
- Removed `accountId` from log metadata (multi-tenant data exposure)
