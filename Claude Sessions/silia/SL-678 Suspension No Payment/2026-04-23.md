- **Goal for this session:** Ensure billing system works retroactively with existing accounts that were created before billing was implemented.

## Work Log (step by step)

### Step 1: Retrocompatibility Analysis
- **What:** Deep exploration of the entire billing module to identify what breaks for pre-billing accounts
- **Why:** Existing accounts have no stripeCustomerId, no billingPlan, no stripeSubscriptionId — needed to map all failure points
- **Files explored:** Billing/application/handlers/StripeWebhookHandler.ts, Billing/infrastructure/aws/handlers/createSubscription/, Billing/infrastructure/aws/handlers/onboardAccount/, Accounts/domain/models/Account.model.ts, Accounts/application/usageFees/getUsageFees.ts, Assistant/domain/models/Chatbot.model.ts
- **Result:** Identified 5 breaking points: missing stripeCustomerId (HIGH), missing billingPlan on Account (HIGH), missing billingPlan on Chatbot (MEDIUM), missing stripeSubscriptionId (MEDIUM), no GSI on stripeCustomerId (LOW)

### Step 2: Design Decision — Unified Flow at Chatbot Level
- **What:** Decided to handle retrocompatibility at the chatbot PUT handler level instead of creating migration endpoints
- **Why:** User's insight — when updating chatbot billing, if no subscription exists, create one. Same endpoint handles both new and existing accounts. No migration scripts needed.
- **Result:** Plan approved — single file change in Assistant/application/Put/index.ts

### Step 3: Implementation — Auto-subscribe on billing plan update
- **What:** Added else-if branch in Assistant/application/Put/index.ts (line ~468) that auto-onboards accounts and creates subscriptions when chatbot has no stripeSubscriptionId
- **Why:** Pre-billing accounts get automatically onboarded + subscribed when someone updates their chatbot's billing plan
- **Files touched:** Assistant/application/Put/index.ts (lines 38, 468-518)
- **Changes:**
  1. Added import: AccountModel from Accounts/domain/models/Account.model
  2. Added else-if branch after existing Stripe sync block:
     - Fetches account via AccountModel.findById(existingChatbot.accountId)
     - If no stripeCustomerId → calls billingService.createCustomer() to onboard
     - Creates subscription via billingService.createSubscription()
     - Saves stripeSubscriptionId and taxRate on chatbot via updateData
  3. Non-blocking (try/catch), same pattern as existing sync code
- **Result:** +51 lines added, compiles cleanly
